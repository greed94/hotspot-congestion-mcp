# 핫플 혼잡도 비서 — MVP 실행 문서 (AGENTIC PLAYER 10)

> 서울 실시간 도시데이터로 "지금/언제 거기 붐비나"를 대화로 답하는 카카오 MCP.
> 이 문서 하나로 바이브코딩 바로 시작 가능. 스타터 코드: `hotspot_mcp.py` (로직 검증 완료).

## 0. 제일 먼저 — 실질 마감은 7/14가 아니라 **7/7**

오늘이 **7/1**입니다. 공모전 접수 마감은 7/14지만, PlayMCP 서버 **심사가 영업일 최대 7일**이고
**7/7(화)까지 심사 요청한 건만 7/10(금)까지 승인 보장**됩니다. 그 이후 요청은 기한 내 승인이 안 될 수 있어요.

→ **진짜 데드라인: 7/7(화)까지 "등록 및 심사 요청" 제출.** 남은 실작업 시간 ≈ 6일.
→ 그래서 MVP는 "얇고 튼튼하게". 기능 욕심 금지.

### 시상 (총 2,700만원)
- 대상 1팀: 과기정통부 장관상 + **1,000만원**
- 은상 2팀: 각 500만원 / 동상 7팀: 각 100만원
- ※ 제세공과금 22% 본인부담

### 참여 방법 (4단계)
1. **카카오클라우드에서 MCP 서버 생성** — Endpoint 생성(인당 2대까지). 공식 노션 가이드: `kko.to/player10`
2. **PlayMCP 개발자 콘솔에 등록** — 테스트 중엔 `임시 등록`, 완성되면 `등록 및 심사 요청`
3. **심사 통과 후 `전체 공개`로 변경** — 기본값이 `나에게만 공개`라, 안 바꾸면 접수 대상에서 **제외됨**
4. **[Player 예선 참여] 버튼으로 최종 제출** — 제출 1회만 가능

### 심사 기준 (예선 = 카카오 내부심사)
- **창의성**: 새 아이디어·파급력  ·  **편의성**: 적합한 UI/UX, 일상의 실질 가치  ·  **안정성**: 안정 구동·정확한 데이터·보안
- 본선 = 내부심사 + 사용자 투표 병행

### 주의
- 카카오클라우드 사용이 원칙(수량 소진 시 없이도 가능, 이 경우 심사 영향 없음)
- 제3자 권리 침해 없어야 함(데이터 출처 적법). 서울 실시간 도시데이터=공공데이터라 OK
- 문의: cs.kakao.com

## 2. MVP 정의 — "핫플 혼잡도 비서"

### 한 줄
서울 주요 핫플의 **지금 혼잡도 + 향후 12시간 예측**을 대화로 답하고, 두 곳을 비교해 더 한산한 곳을 골라준다.

### 왜 MCP여야 하나 (LLM 불가)
"지금 성수 얼마나 붐벼?"는 모델이 절대 모름 → 서울 실시간 데이터를 실시간 호출해야만 답 가능. + 12시간 AI 예측 + 상권 활기.

### 카카오맵 AI 메이트와 차별화 (안 겹침)
- 카카오맵 AI 메이트 = **맛집/장소 추천**("어디 갈까"). 혼잡도는 안 함.
- 우리 = **"언제/지금 붐비나"**(타이밍). 질문 자체가 다름. → "어디 갈지는 카카오맵, 언제 가면 한산한지는 이 도구".

### 도구 (예선 MVP = 딱 4개, 코드 검증 완료)
| 도구 | 하는 일 |
|---|---|
| `get_hotspot_congestion(place)` | 그 핫플 지금 혼잡도(4단계) + 상권 활기 + 한줄평 |
| `compare_hotspots(a, b)` | 두 핫플 현재 혼잡도 비교 → 더 한산한 곳 |
| `best_time_to_go(place)` | 12시간 예측에서 가장 한산한 시간대 TOP3 |
| `list_supported_hotspots()` | 지원 장소 목록 안내 |

### 사용자 흐름 (카톡 대화 예시)
- "성수 지금 붐벼?" → 혼잡도 카드
- "성수랑 홍대 어디가 한산해?" → 비교 + 추천
- "성수 오늘 언제 가면 한산해?" → 예측 TOP3
- (본선 단계에서) 결과를 **스샷 떠서 단톡방 공유** → 유입 루프

### 데이터 · 기술 스택
- 데이터: **서울 실시간 도시데이터 citydata** (서울 열린데이터광장, 무료 인증키)
  - 실시간 인구 혼잡도 4단계(여유/보통/약간 붐빔/붐빔) + 향후 12시간 예측 + 상권(신한카드, 82곳) + 교통/날씨
  - 5분 갱신 · **1회 1개 장소** 호출 · 주요 약 120곳
- 서버: Python + **FastMCP**(mcp SDK) + httpx, 전송 `streamable-http`(배포 전송은 공식 가이드에 맞춰 확정)
- 배포: **카카오클라우드** MCP 서버 endpoint
- 권장: **서버 측 5분 TTL 캐시** (rate limit 회피 + 응답 속도)

### 정직하게 짚을 한계 (문구로 처리)
- 통신사 기지국 **추정치** → "참고" 문구 필수(실제와 오차 가능)
- **서울 주요 120곳만** (우리 집앞 카페 X). "핫플/상권/역세권 혼잡도"로 범위 명확히.
- 상권은 82곳만 / 예측·보조 피드가 없는 장소가 있을 수 있음 → null 처리 필수

### 범위 경계
- **예선**: 위 4개 도구의 표준 MCP 서버 (이 문서 범위)
- **본선(통과 후 7/30~8/27)**: Kakao Tools **위젯 스펙** + 더 엄격한 MCP 표준으로 업그레이드 (예선 후 대응)

## 3. 스타터 코드 (`hotspot_mcp.py`)
- FastMCP 서버 + `fetch_citydata()` + 파서(순수함수) + 도구 4개.
- 응답이 dict/list 어느 형태로 와도 처리하도록 방어 코딩.
- **이 환경에서 검증한 것**: 서버 구동·도구 4개 등록·파싱/비교/예측 로직(합성 데이터) → 전부 통과.
- **아직 검증 못 한 것**: 실제 서울 API 호출(이 샌드박스는 서울 API 도메인 차단). → 아래 T2가 1순위.

## 4. 테스트 체크리스트

### (A) 여기서 이미 검증됨 ✅
- [x] MCP 서버 구동 + 도구 4개 등록(FastMCP `list_tools`)
- [x] `parse_congestion` (list/dict 두 형태 모두)
- [x] `parse_forecast` 시간대 추출
- [x] `pick_best_times` 한산 순 정렬(여유→보통→약간붐빔)
- [x] `compare` 혼잡도 판정 로직

### (B) 바이브코딩에서 **제일 먼저** 테스트 (순서대로) ⚠️
- [ ] **T1. 인증키 발급** — 서울 열린데이터광장(data.seoul.go.kr) 가입 → citydata 인증키(무료). `SEOUL_API_KEY` 주입.
- [ ] **T2. 실제 API 호출 & 필드명 대조 (★최우선★)** — `citydata`를 실제로 호출해 응답 JSON을 눈으로 확인. 코드의 필드명(`AREA_CONGEST_LVL`, `AREA_CONGEST_MSG`, `FCST_PPLTN`, `FCST_TIME`, `FCST_CONGEST_LVL`, `LIVE_CMRCL_STTS`/`AREA_CMRCL_LVL`)이 **실제 응답과 일치하는지** 확인하고 다르면 파서 수정. *여기서 틀어지면 전부 안 돌아감.*
- [ ] **T3. 장소명 정확 표기** — 공식 "주요 120장소 목록(코드포함).xlsx" 로드. 조회는 **정확한 장소 문자열**이어야 함(예: `성수카페거리`, `홍대 관광특구`). `SUPPORTED_PLACES`를 실제 목록으로 교체 + 유저 입력 매칭(오타/별칭 보정).
- [ ] **T4. 빈 값 처리** — 상권(82곳만)·예측(FCST_PPLTN) 없는 장소일 때 크래시 없이 "정보없음" 반환되는지.
- [ ] **T5. rate limit + 캐시** — 개발계정 일일 트래픽 한도 확인. 5분 TTL 서버 캐시 넣어 유저 요청마다 직접 호출 안 하도록.
- [ ] **T6. 보조 피드 장애 대비** — 지하철 도착 등 일부 피드가 일시 중단돼도(공지 이력 있음) 혼잡도 기능은 정상 동작하도록 분리.

### (C) 배포 · 등록 (심사 요청 전)
- [ ] **T7. 카카오클라우드 배포** — 공식 노션(kko.to/player10)대로 MCP 서버 생성·배포, endpoint 확인.
- [ ] **T8. PlayMCP 임시 등록 → AI 채팅 테스트** — 도구 4개가 실제 대화에서 호출되는지. (임시 등록 상태에선 심사 요청 X)
- [ ] **T9. 등록 및 심사 요청 (★7/7까지★)** → 승인 후 `전체 공개` 변경 → [Player 예선 참여] 제출.

## 6. 리스크 & 대비
- **심사 지연**: 7/7 넘기면 위험 → 7/1~3 안에 API 검증까지 끝내는 게 관건.
- **필드명 불일치(T2)**: 가장 흔한 실패. 실제 응답부터 보고 코딩.
- **혼잡도 정확도 논란**: "추정·참고" 문구 + 출처(서울시 실시간 도시데이터) 명시 → 안정성/신뢰 점수 방어.
- **카카오맵 확장 가능성**: 지금 카카오맵 AI 메이트는 혼잡도를 안 하지만 나중에 붙일 수 있음 → "타이밍/예측 + 비교 + 알림" 특화로 각을 좁혀 차별성 유지.
