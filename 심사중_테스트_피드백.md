## 결론

현재 심사 중 서버는 제출 상태로 유지해도 됩니다.

테스트 결과상 핵심 도구 5종은 실제 PlayMCP AI 채팅에서 모두 호출됐고, 치명적인 실패는 보이지 않습니다.

- `get_hotspot_congestion`: 정상
- `compare_hotspots`: 정상
- `best_time_to_go`: 정상
- `recommend_less_crowded_hotspots`: 정상
- `list_supported_hotspots`: 정상

미지원 장소 안내도 정상적으로 동작했습니다. 심사 중에는 재배포나 코드 수정 없이 테스트 결과만 기록하는 것이 좋습니다.

## 잘 된 점

### 자연어 장소 입력

다음 입력들이 의도대로 처리됐습니다.

- `홍대입구역 롯데리아 앞` -> `홍대입구역`
- `연트럴파크` -> `연남동`
- `고터` -> `고속터미널역`
- `강남역 스타벅스` -> `강남역`

- 역세권

### 정상 미지원 처리

다음은 실패가 아니라 정상 동작입니다.

- `부산 해운대 지금 붐벼?`

## 관찰된 아쉬운 점

### 1. 일부 목록 질문은 도구 호출 로그 없이 답변됨

42~44번, 90~92번은 직전 목록 응답을 AI가 대화 맥락으로 재사용한 것으로 보입니다. 실제 기능 문제는 아니지만, 도구 호출 검증 목적이라면 새 채팅에서 한 문장씩 다시 테스트하는 것이 더 정확합니다.

사후 개선 후보:
- 없음. 서버 문제라기보다 AI 채팅의 대화 맥락 재사용입니다.

### 2. `고궁이나 명소`가 `관광특구`로 라우팅됨

37번:

```text
지금 갈 만한 고궁이나 명소 추천해줘
```

이 질문에서 AI가 `category: 관광특구`로 호출했고, 결과가 `잠실 관광특구`로 나왔습니다. 도구 호출은 성공했지만, 사용자 의도는 `고궁·문화·명소`에 더 가깝습니다.

심사 중 영향:
- 치명적이지 않음
- 다만 대표 데모 문구로는 쓰지 않는 것이 좋음

사후 개선 후보:
- `recommend_less_crowded_hotspots` 파라미터 설명에 `고궁`, `명소`, `문화` 카테고리 예시를 더 강하게 노출
- 가능하면 MCP input schema에 `category` 설명을 명시
- 서버 내부에서 `category='관광특구'`가 들어와도 원문에 `고궁/명소/문화`가 있으면 보정하는 방식은 현재 구조상 어려움. 원문 전체를 받는 별도 파라미터가 필요할 수 있음

### 3. `시장 말고 조용한 거리`가 전체 추천으로 처리됨

86번:

```text
지금 시장 말고 조용한 거리 쪽으로 추천해줘
```

AI가 `category: 전체`로 호출했고, 결과가 `강서한강공원`으로 나왔습니다. 사용자는 `상권·거리·시장` 중 시장 제외를 기대했을 수 있습니다.

심사 중 영향:
- 낮음
- 질문 자체가 복합 조건이라 현재 MVP 범위에서는 허용 가능

사후 개선 후보:
- `category` 외에 `exclude` 또는 `preference` 파라미터 추가
- 예: `category='상권·거리·시장'`, `exclude='시장'`
- 현재는 도구가 단일 카테고리 추천만 하므로 복합 조건은 AI가 제대로 넣어줘야 함

### 4. `홍대`가 상황에 따라 `홍대 관광특구` 또는 `홍대입구역`으로 들어감

일부 질문에서 AI가 `홍대`를 `홍대입구역`으로 넘겼습니다. 서버가 처리할 수 있으므로 기능 실패는 아닙니다.

심사 중 영향:
- 낮음

사후 개선 후보:
- 도구 설명 또는 등록 설명에서 `홍대 전체 상권은 홍대 관광특구, 역 주변은 홍대입구역`이라고 구분
- 필요하면 별도 alias 전략을 더 명확히 유지

### 5. AI 최종 답변이 서버 원문보다 요약되어 출처/기준시각이 빠질 수 있음

일부 답변에서는 서버가 제공한 기준시각/출처 문구가 AI 최종 답변에서 축약될 수 있습니다.

심사 중 영향:
- 낮음에서 중간
- 도구 호출 상세를 열면 원문 확인 가능하지만, 사용자가 보는 최종 답변에서는 빠질 수 있음

사후 개선 후보:
- 도구 응답 첫 줄에 기준시각과 출처를 더 강하게 배치
- 등록 설명에 `서울시 실시간 도시데이터 기반이며 통신사 기지국 추정치` 문구 유지

## 추가 테스트 권장 문구

아래 문구는 이미 통과한 1~100번 이후, 더 확인해볼 가치가 있는 케이스입니다. 심사 중 서버 수정 목적이 아니라 관찰/기록 목적입니다.

### A. 목록 도구 재호출 확인

새 채팅에서 한 문장씩 넣는 것을 권장합니다.

```text
101. 역세권 지원 장소만 다시 보여줘
102. 한강공원 목록만 알려줘
103. 상권이랑 거리 지원 장소만 알려줘
104. 고궁이나 문화 명소 목록 보여줘
```

기대:
- `list_supported_hotspots` 호출 또는 직전 목록 재사용
- 새 채팅에서 도구 호출이 나오면 가장 깔끔함

### B. 고궁·명소 라우팅 확인

```text
105. 고궁 중 지금 한산한 곳 추천해줘
106. 문화 명소 중 지금 덜 붐비는 곳 추천해줘
107. 경복궁, 창덕궁, 덕수궁 중 어디가 한산해?
108. 북촌한옥마을이랑 경복궁 중 어디가 덜 붐벼?
```

기대:
- 105~106은 `recommend_less_crowded_hotspots`
- 107~108은 `compare_hotspots`

주의:
- 105~106이 `관광특구`로 가면 사후 개선 후보입니다.

### C. 복합 조건 추천 확인

```text
109. 시장 말고 거리 쪽에서 지금 한산한 곳 추천해줘
110. 실내보다는 산책하기 좋은 곳 중 한산한 데 추천해줘
111. 한강공원 말고 공원 중에서 사람 적은 곳 알려줘
112. 데이트하기 좋은 상권 중 덜 붐비는 곳 추천해줘
```

기대:
- 대부분 `recommend_less_crowded_hotspots`

주의:
- 현재 도구는 제외 조건을 명시적으로 처리하지 않으므로 AI가 `전체`로 보내면 부정확할 수 있음

### D. 애매한 장소명 확인

```text
113. 잠실 지금 어디가 덜 붐벼?
114. 홍대 전체랑 홍대입구역 중 어디가 한산해?
115. 동대문이랑 DDP 중 어디가 덜 붐벼?
116. 여의도랑 여의도한강공원 중 어디가 나아?
```

기대:
- `compare_hotspots`
- 동일/인접 권역을 자연스럽게 안내하면 정상

### E. 미지원·경계 입력 확인

```text
117. 샤로수길 지금 붐벼?
118. 종로3가 지금 사람 많아?
119. 을지로입구역 지금 어때?
120. 서울 밖인데 판교 현대백화점 지금 붐벼?
```

기대:
- 지원 장소로 안전하게 매칭되거나 미지원 안내
- 잘못된 장소로 억지 매칭되면 사후 개선 후보

### F. 시간 범위 한계 확인

```text
121. 내일 오전 강남역 언제 가면 한산해?
122. 이번 주말 홍대 언제 가면 덜 붐벼?
123. 지금부터 2시간 안에 성수 가도 괜찮아?
124. 오늘 밤 10시에 여의도한강공원 괜찮아?
```

기대:
- 현재 API는 향후 12시간 예측 중심이므로, `내일/주말` 질문은 한계 안내가 나오면 좋음
- `오늘 밤 10시`처럼 12시간 안이면 `best_time_to_go` 또는 현재 예측 기반 답변

## 심사 중 운영 원칙

심사 요청이 이미 들어간 상태이므로 다음을 권장합니다.

1. 코드 수정, 이미지 재빌드, 서버 재등록은 하지 않기
2. 테스트 중 이상한 답변은 이 문서에 기록만 하기
3. 같은 문항을 새 채팅에서 1개씩 다시 넣어 재현성 확인
4. 실제 장애가 아닌 라우팅/표현 이슈는 본선 또는 심사 후 `v1.0.1`/`v1.1.0` 후보로 남기기

## 사후 수정 후보 우선순위

### P1. 카테고리 라우팅 보강

대상:
- 고궁/명소/문화
- 시장 제외 거리
- 한강 제외 공원

방향:
- 파라미터 설명 강화
- 가능하면 enum 또는 상세 description 제공
- 복합 조건용 파라미터 추가 검토

### P2. 사용자 최종 답변의 기준시각/출처 보존

방향:
- 서버 응답의 첫 줄과 마지막 줄을 AI가 요약해도 살아남도록 문구 구조 조정
- 심사 중에는 수정하지 않음

### P3. 홍대 권역 구분

대상:
- 홍대 관광특구
- 홍대입구역(2호선)
- 연남동

방향:
- 등록 설명 또는 도구 설명에 권역 구분 명시
- 사용자가 `홍대 전체`, `홍대입구역`, `연남`을 구분해서 물으면 더 정확해짐

### P4. 제외 조건 추천

예:
- 시장 말고 거리
- 한강 말고 공원
- 너무 멀지 않은 곳

방향:
- `recommend_less_crowded_hotspots`에 `exclude`/`preference` 파라미터 추가 검토
- 본선 또는 심사 후 개선 후보

## 현재 판단

현재 테스트 결과는 심사 통과를 노릴 수 있는 안정 상태입니다.

# Claude 추가 리뷰 (2026-07-07)

위 문서를 실제 코드와 대조 검증했습니다. 종합 판단은 문서와 같습니다 — **서버 장애·파싱 실패 없음, 50/50 성공, 심사 중 무조작이 맞음.** 아래는 (a) 문서가 놓친 실질 이슈 1건, (b) 개선 방향 정정 1건, (c) 추가 관찰, (d) 추가 테스트 문구입니다. 모두 심사 후/본선용이며 지금 고치는 게 아닙니다.

## ★ 새로 발견 (문서 누락, 실질 이슈): 오타보정이 미지원 장소를 엉뚱한 서울 장소로 억지 매칭

로컬에서 `resolve_place`를 직접 돌려 확인했습니다.

```text
판교역   -> 교대역          (경기 판교인데 서울 교대역 데이터가 조용히 나옴)
샤로수길 -> 가로수길        (서울대입구 앞인데 신사 가로수길로 감)
부산 / 부산 해운대 / 제주공항 / 을지로입구역 -> notfound (이건 정상)
```

**원인**: `resolve_place` 5단계 중 마지막 difflib 오타보정이 `cutoff=0.6`으로 느슨합니다. 근접 후보가 딱 하나면 그대로 확정해버려서, 실제로는 지원하지 않는 장소가 발음이 비슷한 서울 장소로 매칭됩니다.

**왜 중요한가**: 이건 "못 찾음" 안내보다 나쁩니다. 심사위원이 `판교역`을 물으면 **교대역 데이터를 정답인 양 조용히 받습니다.** 심사 기준의 "정확한 데이터·안정성"에 직접 불리하게 작용할 수 있습니다.

**심사 중**: 손대지 않습니다(무조작 원칙).

**본선 수정 방향**:
- difflib `cutoff` 0.6 → 0.72~0.75 상향 (판교역→교대역은 차단되고, `강남녁`처럼 진짜 오타는 유지됨)
- 또는 근접 매칭 시 바로 확정하지 말고 `혹시 '교대역'을 찾으시나요?` 확인형으로 되묻기
- `샤로수길→서울대입구역`, `을지로입구역→(인접 권역)` 등 실제 근처 권역 alias 보강

## 정정: 문서 이슈 #2(고궁/명소 라우팅)의 개선 방향

문서는 원인 진단은 맞지만, 개선 후보 중 **"서버 내부에서 원문 전체를 받는 별도 파라미터가 필요"** 부분은 과설계입니다. 로컬 확인 결과:

```text
_category_key("고궁") / ("명소") / ("문화")  -> 전부 "고궁·문화·명소"
후보: 경복궁, 광화문·덕수궁, 보신각, 서울 암사동 유적, 창덕궁·종묘 ...
```

즉 **코드는 이미 고궁/명소/문화를 정확히 라우팅합니다.** 37번이 관광특구로 샌 건, AI가 "고궁"을 넣을 수 있는지 몰라서(도구 설명 예시에 없음) 목록에 있던 관광특구를 고른 것뿐입니다.

→ 따라서 필요한 건 **도구 설명 예시에 "고궁, 명소" 추가 1줄**이 전부입니다. 별도 파라미터·원문 보정 로직 불필요. (문서 P1의 첫 번째 bullet만 맞고, 세 번째 bullet은 폐기 권장)

## 추가 관찰 (본선 백로그)

1. **반대 의도 미지원 (기능 갭)**: "지금 제일 붐비는 곳"에 대응할 도구가 없습니다(recommend는 한산한 곳만). 축제·핫플 탐색이나 "피할 곳"을 알려는 수요가 있음. → 본선: recommend에 방향(한산/붐빔) 파라미터 또는 `most_crowded` 추가 검토.
2. **compare에 미지원 1곳 섞이면 전체 취소**: `홍대랑 부산 비교` → 부산 못 찾음 안내만 뜨고 홍대 비교도 안 됨. → 유효한 곳만 비교하고 나머지는 "부산은 미지원" 주석 처리하면 더 친절.
3. **best_time 전부-붐빔 정직 문구 미검증**: v1.0.0에 "12시간 내내 붐비는 편이에요 · 그나마 나은 시간대" 문구를 넣었으나, 50문항이 전부 야간(여유)에 걸려 발동 안 됨. 낮 피크 시간대 테스트 필요.
4. **다 붐비는 카테고리서 "여유" 과장**: 36번 잠실 관광특구 5만명을 AI가 "여유롭다"고 표현. 서버 첫 줄은 `비교적 한산한 곳 추천`이라 정직하지만 AI 요약이 과장. → 상위 항목의 실제 등급(약간 붐빔 등)을 더 도드라지게.
5. **limit 상한 8**: "TOP 10 알려줘" 요청해도 최대 8곳까지만 나옴(`min(limit, 8)`). 의도된 동작이나 문서화 가치 있음.
6. **영어 입력 전부 notfound**: `hongdae`, `Gangnam station` 매칭 안 됨. 한국어 서비스라 치명적이지 않지만, 본선에서 글로벌·외국인 관광객을 노린다면 영문 alias 추가 여지.
